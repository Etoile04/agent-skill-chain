# 经验摘要化模板

> 将原始经验转化为策略偏好的标准方法。用于 Learning Writeback 和规划层注入。

## 1. 核心原则

### 1.1 摘要化的目标

将具体的、情境绑定的经验（"我做了 X，因为 Y"）转化为**可复用的、情境无关的策略偏好**（"这类问题通常需要考虑 Y"）。

关键约束：
- **保留决策逻辑**，丢弃具体命令
- **保留错误模式**，丢弃具体错误消息
- **保留判断标准**，丢弃具体数值（除非是阈值）

### 1.2 抽象层级

| 层级 | 描述 | 示例 |
|------|------|------|
| L0: 原始记录 | 具体操作 + 具体结果 | "运行 `npm install` 失败，报 EACCES 权限错误" |
| L1: 模式识别 | 提取通用模式 | "npm 全局安装需要检查权限" |
| L2: 策略偏好 | 转化为推荐做法 | "包管理操作前检查当前用户权限和目标目录写权限" |
| L3: 元策略 | 跨领域通用规则 | "任何安装操作前先验证环境前提条件" |

摘要化目标是 **L1-L2**。L3 过于抽象（"做事要小心"），L0 过于具体。

## 2. 转化规则

### 规则 1: 命令 → 策略

| 原始 | 转化方法 | 策略偏好 |
|------|---------|---------|
| "用 `curl -X POST` 调 API 返回 403" | 去掉具体命令，保留操作类型和失败原因 | "外部 API 调用需要先验证认证状态" |
| "运行 `python -m pytest` 超时" | 去掉具体工具，关注操作属性 | "长时间运行的测试套件需要考虑超时策略" |
| "编辑 config.json 后服务崩溃" | 去掉具体文件，关注配置变更的影响范围 | "配置变更需要考虑服务重启和向后兼容" |

### 规则 2: 错误 → 模式

| 原始 | 转化方法 | 模式 |
|------|---------|------|
| "ECONNREFUSED on port 8080" | 提取连接失败模式 | "网络连接类错误首先检查目标服务是否启动" |
| "Permission denied: /root/.ssh" | 提取权限模式 | "文件访问错误首先检查路径权限和当前用户" |
| "OOM killed (node, 4GB)" | 提取资源限制模式 | "大数据处理任务需要评估内存需求并分批处理" |

### 规则 3: 成功 → 方法论

| 原始 | 转化方法 | 方法论 |
|------|---------|--------|
| "先跑测试再提交，避免了 3 个 bug" | 提取验证顺序的价值 | "变更后先运行验证步骤再推进" |
| "分步调试发现是第 2 步的问题" | 提取调试策略 | "复杂问题分步隔离，逐段验证" |
| "查了文档才知道参数格式变了" | 提取信息获取策略 | "API 集成前先确认版本和接口文档" |

### 规则 4: 保留例外

以下信息**不应过度抽象**，需要保留具体细节：

| 类型 | 为什么保留 | 示例 |
|------|-----------|------|
| 具体的阈值/限制 | 是硬性约束 | "飞书 API 限流: 每秒 50 次" |
| 特定版本的已知 bug | 无法从 bug 推断 | "acpx 0.5.3 不支持 --cwd 选项" |
| 特定平台的 workaround | 平台特定 | "macOS 的 sed 需要 -i '' 参数" |
| 安全相关约束 | 过度抽象会丢失风险 | "文件路径必须在 workspace 目录下" |

## 3. 转化示例

### 示例 1: 正确转化

**原始经验**:
> "用 acpx codex exec --cwd /path 'prompt' 失败了，报 unknown option '--cwd'。后来发现 --cwd 是全局选项不是子命令选项，改成 cd /path && acpx --format quiet codex exec 'prompt' 就好了。"

**转化过程**:
1. 去掉具体命令 → "CLI 工具的选项可能属于不同层级（全局 vs 子命令）"
2. 提取模式 → "命令行参数位置不匹配是常见的 CLI 陷阱"
3. 保留例外 → "acpx 的 --cwd 和 --format 是全局选项"

**策略偏好（L2）**:
```
CLI 工具使用前先验证参数层级（全局选项 vs 子命令选项）。
可用 <tool> <subcommand> --help 确认参数位置。
已知例外：acpx 的 --cwd、--format 是全局选项。
```

### 示例 2: 正确转化（错误场景）

**原始经验**:
> "OntoFuel 任务 4 并发触发 API 限流，延迟超过 12 小时。后来降到 2 并发加了 5 分钟间隔才恢复。"

**转化过程**:
1. 去掉具体项目名 → "批量 API 调用任务的并发控制"
2. 提取模式 → "高并发会触发 API 限流，需要根据 API 限制调整"
3. 提取可复用阈值 → "并发 2 + 间隔 5 分钟是安全的默认值"

**策略偏好（L2）**:
```
批量 API 调用任务需要控制并发和间隔。
默认建议：并发数 2，批次间隔 5 分钟。
遇到限流时先降低并发，而非增加重试。
```

### 示例 3: 过度抽象（错误示范）

**原始经验**:
> "飞书文件发送失败，因为文件在 /tmp 下。飞书要求文件在 workspace 目录下才行。移到 workspace 后发送成功。"

❌ **过度抽象（L3）**:
```
做事前先检查规则。
```
（过于空泛，无法指导具体行动）

❌ **保留过多原始细节（L0）**:
```
发送 /tmp/test.md 到飞书失败，改用 /Users/lwj04/.openclaw/workspace/reports/test.md 成功。
错误信息: LocalMediaAccessError: Local media path is not under an allowed directory。
```
（过于具体，换了文件名就无法复用）

✅ **正确的策略偏好（L2）**:
```
飞书文件发送需要文件路径在 workspace 允许目录下。
允许: /Users/lwj04/.openclaw/workspace/ 及子目录。
禁止: /tmp、/var、/Downloads 等外部目录。
发送前用绝对路径验证文件位置。
```

## 4. 摘要化粒度指南

### 何时抽象

| 信号 | 动作 |
|------|------|
| 同类错误出现 2 次以上 | 抽象为模式，记录到 planning_hints |
| 成功方法可复用到其他任务 | 抽象为策略，考虑生成技能卡 |
| 调试过程超过 3 步 | 提取调试方法论，而非具体步骤 |
| 涉及特定工具/平台 | 抽象策略 + 保留具体例外 |

### 何时保留细节

| 信号 | 动作 |
|------|------|
| 仅出现过 1 次的偶发问题 | 保留原始记录，不急于抽象 |
| 涉及特定版本/平台的 bug | 完整保留版本号和 workaround |
| 安全相关约束 | 保留完整约束条件 |
| 具体的数值阈值 | 保留原始数值，不过度泛化 |

### 摘要化检查清单

对每条原始经验，问以下问题：

- [ ] 去掉具体命令/文件名后，策略是否仍然有意义？
- [ ] 策略是否可以应用到**同一类**的其他问题？
- [ ] 是否保留了必要的例外/约束信息？
- [ ] 抽象层级是否在 L1-L2（不是 L0 也不是 L3）？
- [ ] 是否有可执行的检查项（"做 X 前先检查 Y"）？

## 5. 与系统的集成

### 输入来源

```
StepResult (status=failed)
  → 自动提取错误模式
  → 按 .learnings/TEMPLATE.md 格式记录

StepResult (status=success, 有策略价值)
  → 生成 draft 技能卡到 skill-cards/pending/
  → 等待人工审核后移至正式目录

memory/YYYY-MM-DD.md (原始日志)
  → 可定期批量摘要化
  → 使用 extract-skill-card.sh 从日志中提炼
```

### 输出去向

```
策略偏好 → planning_hints（注入 Planning Agent）
错误模式 → planning_hints.common_failures
方法论 → skill-cards 的 e_workflow
具体约束 → skill-cards 的 e_mistake（保留细节）
```

### 自动 vs 人工

| 步骤 | 自动 | 人工 |
|------|------|------|
| 错误模式提取 | ✅ | 审核分类 |
| ERRORS.md 追加 | ✅ | 确认 5-Why 根因 |
| Draft 技能卡生成 | ✅ | 审核 + 移至正式目录 |
| 日志追加 | ✅ | — |
| 策略偏好更新 | 建议 | 确认 + 执行 |
