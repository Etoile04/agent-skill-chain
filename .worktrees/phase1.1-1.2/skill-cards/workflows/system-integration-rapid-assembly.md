# 系统整合快速组装工作流

## ✅ 成功经验 (e_success)

### 经验传递分级
```yaml
experiences:
  direct: []  # 无硬性领域约束
  indirect:
    - "分层设计 + 单向流动 + 自动化验证 + 一次完成"
    - "趁热打铁，一次完成到 100%，避免上下文切换"
    - "单向数据流是整合成功的关键约束"
  forbidden:
    - "具体脚本和文件路径见 e_workflow"
```

### 有效的策略
- **分层设计**：每个系统专注自己的强项（memory/ 原始轨迹 → skill-cards/ 结构化摘要 → .learnings/ 错误明细）
- **单向流动**：信息只流动一次，避免双向同步的复杂性
- **自动化验证**：测试脚本确保系统正常，100% 通过率
- **趁热打铁**：立即完成 100%，避免遗忘和上下文丢失

### 关键决策
- **2 小时完成整合**：审计 30min → 设计 30min → 实施 1h → 测试即时
- **测试优先设计**：先写测试再实施，确保可验证

### 验证过的工具/方法
- 记忆系统整合：测试通过率 100%（5/5），数据流完整度 100%（4/4）
- 自动化测试脚本：`scripts/test_memory_flow.py`

## ❌ 失败教训 (e_mistake)

### 踩过的坑
- **分多次做效率极低**：上下文切换成本高，容易遗忘前次决策
- **双向同步导致数据不一致**：多系统双向写入产生冲突
- **没有自动化测试的整合不可靠**：手动测试覆盖不全

### 错误模式
- **拖沓整合**：症状（整合拖多天）→ 根因（分多次做 + 上下文切换）→ 修复（一次完成）
- **双向同步冲突**：症状（数据不一致）→ 根因（多系统双向写入）→ 修复（单向数据流）

### 需要避免的做法
- 不要把整合项目分多天做
- 不要设计双向数据同步
- 不要省略自动化测试

## 🔄 推荐工作流 (e_workflow)

### 标准流程
1. **审计现状**（30min）— 盘点所有涉及的系统、接口、数据流
2. **设计方案**（30min）— 分层 + 单向流动 + 接口定义
3. **快速实施**（1h）— 趁热打铁，一次完成核心功能
4. **自动化测试**（即时）— 每个接口都有测试用例
5. **验收**（15min）— 确认 100% 通过率 + 文档更新

### 触发条件
- 需要整合 2+ 个子系统
- 系统间需要数据流动
- 有重复的手动操作需要自动化

### 前置条件
- 了解所有涉及的系统
- 有实施权限
- 有测试环境

### 预期结果
- 整合完成度 100%
- 测试通过率 100%
- 数据流完整

### 回退方案
- 整合失败 → 回退到整合前状态（审计时已记录）
- 测试不通过 → 逐个接口排查
- 时间不够 → 至少完成核心接口 + 测试，其余标记 TODO
