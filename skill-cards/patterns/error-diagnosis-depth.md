# 错误诊断深度不足模式

## ✅ 成功经验 (e_success)

### 经验传递分级
```yaml
experiences:
  direct: []  # 无硬性领域约束
  indirect:
    - "遇到错误先分类（表面/中间/根本），用 5-Why 追到根因，再制定修复方案"
    - "暂停后不会自动恢复，必须主动检查和干预"
    - "看到 timeout 先查 Runtime 值，负数 = 未启动 = 系统级问题"
  forbidden:
    - "具体诊断命令和脚本见 e_workflow"
```

### 有效的策略
- **5-Why 分析法**：连续追问 5 层为什么，追到根本原因
- **先分类后行动**：错误分为表面/中间/根本三层，先判断在哪一层
- **主动恢复机制**：每 30 分钟健康检查 + 自动重启失败任务

### 关键决策
- **看到 timeout 先查 Runtime**：发现负数 Runtime → 判断启动失败 → 检查配额
- **不等自动恢复**：暂停后立即干预，设置超时告警

### 验证过的工具/方法
- 5-Why 分析：从 "timeout" → "API 限流" → "配额耗尽" → "并发过高"
- 健康检查脚本：每 30 分钟自动检测任务状态

## ❌ 失败教训 (e_mistake)

### 踩过的坑
- **调整 timeout 配置解决不了启动失败**：Runtime 负数说明任务根本没启动，不是执行超时
- **假设暂停后会自动恢复**：OntoFuel 任务暂停后无限期等待，直到人工干预
- **只修表面不修根因**：重复调整 timeout 而没有检查系统资源

### 错误模式
- **浅层诊断**：症状（timeout）→ 错误修复（调 timeout）→ 无效 → 继续失败
- **正确诊断**：症状（timeout）→ 检查 Runtime → 发现负数 → 判断启动失败 → 检查系统资源/配额

### 需要避免的做法
- 不要看到 timeout 就直接调整 timeout 配置
- 不要假设暂停/失败后系统会自动恢复
- 不要停留在表面错误信息

## 🔄 推荐工作流 (e_workflow)

### 标准流程
1. **记录表面现象** — 完整记录错误信息、时间、上下文
2. **错误分类** — 判断是表面/中间/根本哪一层的问题
3. **5-Why 分析** — 连续追问直到找到根本原因
4. **制定修复方案** — 针对根本原因修复，不治标不治本
5. **预防规则** — 写入 .learnings/ERRORS.md，防止重复

### 触发条件
- 遇到重复出现的错误
- 错误修复后仍然出现
- 任务出现负数 Runtime 或超时

### 前置条件
- 有完整的错误日志
- 了解系统架构和数据流

### 预期结果
- 找到根本原因
- 修复后不再重复出现
- 预防规则已记录

### 回退方案
- 5-Why 分析到第 3 层仍无结论 → 换个角度（从系统资源/网络/权限切入）
- 修复后仍失败 → 检查是否遗漏了其他依赖错误
- 根本原因不明确 → 先做临时缓解，标注待深入分析
